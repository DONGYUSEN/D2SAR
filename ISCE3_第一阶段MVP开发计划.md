# ISCE3 第一阶段 MVP 开发计划

> 目标：以最小可行范围验证“围绕 `isce3 / dolphin / plant-isce3` 开发外围系统”的路线可行性。

---

## 1. MVP 目标

第一阶段不追求全传感器、全流程、全交付，而只验证三件事：

1. 第三方数据能否被稳定适配到统一外部数据契约；
2. 外围 CLI 编排器能否稳定调用上游算法完成一条完整处理链；
3. 输出结果能否以标准格式导出并用于后续查看与分析。

---

## 2. MVP 范围建议

### 2.1 只选一个传感器

建议优先选：

- **GF3**（推荐，如果样例数据最容易获取）

原因：

- 文档中已自然以 GF3 作为例子；
- 便于先完成一个 reader；
- 可避免多传感器并行导致适配层复杂化。

### 2.2 只选一条主流程

建议第一阶段优先做：

- **RTC** 或 **InSAR** 二选一

推荐顺序：

1. 若目标是先验证“单景产品标准化与 geocode/rtc 能力”，优先做 **RTC**；
2. 若目标是尽快逼近核心业务价值，优先做 **InSAR**。

更保守的建议是：

> **先做 RTC MVP，再做 InSAR MVP。**

因为 RTC 对主从数据一致性要求更低，更适合作为适配层与编排层的第一验证点。

### 2.3 第一阶段不做的内容

第一阶段明确不做：

- 多传感器统一调度
- Web 前端
- 多用户并发
- 复杂任务队列系统
- DInSAR 全改正链（ionosphere / troposphere / SET 全开）
- 大规模生产化部署

---

## 3. 第一阶段交付物

## 3.1 必须交付

### 交付物 A：一个第三方 reader

例如：

- `gf3_reader.py`

职责：

- 读取 GF3 原始/产品文件；
- 提取 metadata；
- 生成符合外部数据契约的数据包；
- 输出 `manifest.json`。

### 交付物 B：一个 CLI 编排器

例如：

- `sar_pipeline.py`

职责：

- 接收标准化数据包；
- 调用 `plant-isce3` / `isce3` / 必要时 `dolphin`；
- 支持分步执行与全流程执行；
- 输出日志与结果目录。

### 交付物 C：基础辅助模块

例如：

- `dem_manager.py`
- `orbit_manager.py`

职责：

- 下载 / 缓存 / 索引 DEM 与轨道文件；
- 提供统一路径解析与读取能力。

### 交付物 D：结果导出模块

例如：

- `export_geotiff.py`
- `export_png.py`

职责：

- 将主要处理结果转换为 GeoTIFF / PNG；
- 为后续可视化与 Web 展示做准备。

## 3.2 可选交付

- Dockerfile
- 一份示例配置 YAML
- 一份端到端运行说明文档

---

## 4. 第一阶段推荐处理链

## 4.1 方案 A：RTC MVP（推荐作为第一步）

```text
GF3 原始/产品数据
  ↓
gf3_reader
  ↓
标准化数据包（manifest + slc/orbit/dem/metadata）
  ↓
CLI 编排器
  ↓
调用 plant-isce3 / isce3 RTC 相关能力
  ↓
生成 RTC 结果
  ↓
导出 GeoTIFF / PNG
```

优点：

- 链路相对短；
- 对双景一致性要求较低；
- 容易验证适配层正确性。

## 4.2 方案 B：InSAR MVP

```text
GF3 master/slave
  ↓
gf3_reader
  ↓
标准化双景数据包
  ↓
CLI 编排器
  ↓
配准 / crossmul / filter
  ↓
输出 RIFG 类结果
  ↓
导出 GeoTIFF / PNG
```

优点：

- 更接近核心业务价值；
- 能尽早暴露主从数据适配问题。

缺点：

- 对 metadata 和几何一致性要求更高；
- 更容易在早期被数据问题卡住。

---

## 5. 推荐开发顺序

### 阶段 1：定义契约

- 固化外部数据契约
- 固化目录结构
- 固化 manifest 字段

### 阶段 2：实现 reader

- 完成一个传感器 reader
- 用真实样例数据验证
- 明确 `manifest` 路径语义：`metadata/*` 相对路径，外部源数据与 DEM 使用绝对路径或结构化 ZIP 路径

### 阶段 3：实现辅助模块

- DEM 管理
- Orbit 管理
- 路径缓存与索引

### 阶段 4：实现 CLI 编排器

- 单步执行
- 全流程执行
- 日志与错误输出
- 统一按 `manifest.json` 解析相对 `metadata` 路径与 ZIP 结构化路径

### 阶段 5：实现导出模块

- GeoTIFF
- PNG
- 可选 KML

### 阶段 6：Docker 化

- 固定环境
- 固定依赖
- 固定运行方式

Web 不进入第一阶段。

---

## 6. 模块边界建议

## 6.1 reader 层

输入：

- 第三方原始文件 / 产品文件

输出：

- 标准化数据包

不负责：

- 直接调算法细节
- 导出展示格式

## 6.2 orchestration 层

输入：

- 标准化数据包
- 流程配置

输出：

- 中间结果目录
- 最终产品目录

不负责：

- 解析原始传感器私有格式

## 6.3 export 层

输入：

- 处理结果

输出：

- GeoTIFF / PNG / KML

不负责：

- 核心算法计算

---

## 7. 成功标准

第一阶段完成的判据建议定义为：

1. 能够用一套标准命令处理一个真实样例数据集；
2. 能够稳定生成至少一类标准结果产品；
3. 结果能够导出为 GeoTIFF 和 PNG；
4. 环境能够通过 Docker 复现；
5. 全过程日志可追踪、输入输出路径清晰。

---

## 8. 风险控制

### 风险 1：reader 迟迟无法稳定

应对：

- 缩小到一个传感器、一个模式、一个样例先跑通。

### 风险 2：CLI 编排器过早复杂化

应对：

- 第一版只支持线性流程，不做复杂任务图。

### 风险 3：过早引入 Web

应对：

- Web 不纳入第一阶段验收。

### 风险 4：上游依赖频繁变化

应对：

- 固定 Docker 镜像版本；
- 固定最小支持版本矩阵。

---

## 9. 建议的第一阶段产出顺序

如果按实际落地顺序排，应为：

1. `ISCE3_外部数据契约草案.md`
2. 选定传感器与样例数据
3. `gf3_reader.py`（或选定传感器 reader）
4. `dem_manager.py` / `orbit_manager.py`
5. `sar_pipeline.py`
6. 结果导出脚本
7. Dockerfile

---

## 10. 小结

第一阶段的目标不是把完整系统做完，而是验证这条路线：

> **第三方数据 → 标准化契约 → CLI 编排 → 上游算法调用 → 标准结果导出**

只要这条链路对一个传感器、一条流程跑通，后续再扩展到多传感器、多流程、Docker、Web，风险就会低很多。
